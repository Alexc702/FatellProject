# FaceXiang v1 Async Flow & Fallback Spec

## 1. 目标

本文件用于保护首个 `first aha`，避免用户在第一次体验中被：

- 长时间等待
- 模型超时
- 图像质量不足
- 含糊错误提示

直接劝退。

## 2. 设计原则

### 2.1 first aha 不能被黑盒等待打断

用户第一次自拍后，必须尽快知道系统正在做什么、还需要多久、如果失败该怎么做。

### 2.2 失败要具体，不要抽象

不要只说“识别失败”。  
必须告诉用户：

- 为什么失败
- 该怎么重试

### 2.3 可降级，不硬崩

如果完整结果暂时拿不到，优先考虑：

- 重试
- 降级版结果
- 明确回退

## 3. 首次会话状态机

建议状态如下：

1. `idle`
2. `photo_selected`
3. `quality_checking`
4. `quality_failed`
5. `uploading`
6. `analyzing`
7. `readme_partial_ready`
8. `readme_ready`
9. `context_collecting`
10. `today_ready`
11. `failed`

## 4. 首次路径时延预算

v1 建议目标：

- 图片质量判断：`< 1s`
- 上传：`< 2s` 常规网络
- 特征抽取 + 规则映射：`< 6s`
- 完整 `Read Me`：`8-12s`

如果超过：

- 进入超时提示和降级策略

## 5. 质量门失败回退

## 5.1 失败类型

- 没有检测到脸
- 多人
- 非正脸
- 过暗
- 过糊
- 遮挡严重

## 5.2 用户提示要求

每种失败必须有：

- 明确原因
- 一句重拍建议
- 一个重新拍照按钮

## 6. 分析中体验

分析中页面必须告诉用户：

- 系统正在做什么
- 这是为了什么
- 预计还要一点时间

可展示的阶段文案：

- 正在检查照片质量
- 正在识别面部特征
- 正在整理你的第一版画像

## 7. 超时与失败策略

## 7.1 质量门通过但模型超时

策略：

- 自动重试一次
- 超过阈值后展示友好失败页

失败页要求：

- 不是技术报错
- 提供再次尝试
- 不让用户怀疑图片已丢失

## 7.2 结构化输出不合法

策略：

- 自动要求模型重试一次
- 如仍失败，进入失败页

## 7.3 网络异常

策略：

- 明确提示网络问题
- 保留本地照片选择状态
- 支持再次提交

## 7.4 降级结果

v1 可选降级：

- 如果完整结果暂不可用，可先给简版 `Read Me`
- 只包含：
  - 第一印象
  - 一个主 tension 假设
  - 一个后续问题

前提：

- 必须仍然来自结构化 trace
- 不能用 generic 安慰文案滥填

## 8. Today 与 Ask 的异步策略

### Today

- 进入页面即请求
- 如果失败，展示“今天的内容还在整理中”
- 支持手动刷新

### Ask

- 问题提交后先展示“正在沿着这个主题继续整理”
- 如果超时，保留用户输入内容

## 9. 埋点建议

必须记录：

- 质量门失败原因分布
- 分析中超时率
- 自动重试成功率
- 首个 Read Me 完成率
- 失败后再次尝试率

## 10. 测试重点

### 10.1 异常网络

- 弱网
- 断网后重试

### 10.2 非法输出

- JSON 不合法
- 缺字段
- 置信度极低

### 10.3 极端图片

- 侧脸
- 模糊
- 多人
- 遮挡

## 11. Phase 1 验收门槛

至少满足：

- 质量失败时，用户知道怎么重拍
- 超时不会变成死等待
- 失败后用户能一键再次尝试
- 分析中体验不会让用户误以为 app 卡死
